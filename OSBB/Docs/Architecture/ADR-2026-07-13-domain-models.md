## Контекст

В рамках создания нового слоя `core_new/` необходимо определить набор доменных моделей, которые будут представлять основные сущности системы.

Эти модели должны:
1. Быть **независимыми** от интерфейсов (Telegram, Streamlit, API)
2. Содержать **бизнес-логику**, а не просто данные
3. Быть **тестируемыми** изолированно
4. Обеспечивать **постепенный переход** от старого кода

---

## Решение

Создать четыре основные доменные модели, каждая из которых представляет ключевую сущность системы OSBB.

### 1. Vehicle (Автомобиль)

**Назначение:** Управление автомобилями и их тарифами.

**Ключевые концепции:**
- Автомобиль всегда принадлежит квартире (кроме стадии кандидата)
- Тариф (`parking_time`) определяет режим парковки: Day, Night, Inactive, NULL
- Жизненный цикл: `кандидат` → `создан` → `активен` → `архивирован`

**Методы:**
- `get_by_id(id)` — получение по ID
- `get_by_apartment(apartment)` — все авто квартиры
- `set_parking_time(status)` — установка тарифа
- `update_plate(new_plate)` — изменение номера
- `update_model(new_model)` — изменение марки

**Свойства:**
- `display_name` — "AA8098MM | Toyota"
- `status_display` — "☀️ День" / "🌙 Ночь" / "🚫 Не паркуется"
- `is_parking` — паркуется ли (Day или Night)
- `is_active` — активен ли (не Inactive)

**Решение о кандидате:**  
Создана отдельная сущность `VehicleCandidate` для автомобилей, которые ещё не привязаны к квартире. Это позволяет обрабатывать заявки от жителей до их подтверждения оператором.

---

### 2. Resident (Житель)

**Назначение:** Управление пользователями Telegram и их связью с квартирами.

**Ключевые концепции:**
- Житель идентифицируется по `telegram_user_id`
- Может быть привязан к одной квартире
- Статус определяет уровень верификации
- Роль определяет доступные функции

**Статусы:**
| Статус | Описание |
|--------|----------|
| `new` | Новый пользователь, квартира не привязана |
| `apartment_confirmed` | Самостоятельно подтвердил квартиру |
| `operator_verified` | Проверен оператором |
| `blocked` | Заблокирован |

**Роли:**
| Роль | Описание |
|------|----------|
| `resident` | Обычный житель |
| `admin` | Администратор |
| `super_admin` | Супер-админ |
| `guard` | Охрана |

**Методы:**
- `get_by_telegram_id(id)` — получение по Telegram ID
- `get_by_apartment(apartment)` — все жители квартиры
- `get_all(limit)` — список всех жителей
- `link_apartment(apartment)` — привязать квартиру
- `unlink_apartment()` — отвязать квартиру
- `mark_verified_by_operator()` — подтвердить оператором
- `get_vehicles()` — автомобили жителя

---

### 3. Apartment (Квартира)

**Назначение:** Объединение жителей, автомобилей и статусов согласования.

**Ключевые концепции:**
- Квартира идентифицируется по номеру (`apartment_number`)
- Может иметь множество жителей и автомобилей
- Статус согласования определяет, подтверждена ли квартира

**Статусы согласования:**
| Статус | Описание |
|--------|----------|
| `new` | Не согласована |
| `confirmed` | Согласована |
| `deferred` | Отложена |
| `conflict` | Конфликт |
| `in_progress` | В работе |

**Методы:**
- `get_by_number(apartment)` — получение по номеру
- `get_all(limit)` — все квартиры
- `get_by_entrance(entrance)` — квартиры подъезда
- `format_card()` — карточка для отображения
- `get_summary()` — сводка по квартире

**Свойства:**
- `display_name` — "🏠 105 (п. 1)"
- `residents` — список жителей
- `vehicles` — список автомобилей
- `residents_count` — количество жителей
- `vehicles_count` — количество автомобилей
- `verification_status` — статус согласования

**Решение о кандидате:**  
Создана отдельная сущность `ApartmentCandidate` для квартир, которые ещё не созданы в БД, но появляются в источниках (telegram, tbot).

---

### 4. Payment (Платёж)

**Назначение:** Управление финансовыми операциями.

**Ключевые концепции:**
- Платёж всегда связан с квартирой (или автомобилем)
- Имеет статус, определяющий жизненный цикл
- Хранит информацию об услуге и типе оплаты

**Статусы:**
| Статус | Описание |
|--------|----------|
| `pending` | Ожидает |
| `completed` | Завершён |
| `failed` | Ошибка |
| `refunded` | Возврат |
| `cancelled` | Отменён |

**Типы платежей:**
| Тип | Описание |
|-----|----------|
| `cash` | Наличные |
| `card` | Карта |
| `bank` | Безналичный |
| `transfer` | Перевод |

**Методы:**
- `get_by_id(id)` — получение по ID
- `get_by_apartment(apartment)` — платежи квартиры
- `get_by_vehicle(vehicle_id)` — платежи по авто
- `get_by_service(service)` — платежи по услуге
- `get_by_date_range(start, end)` — платежи за период
- `create(...)` — создание платежа
- `complete()` — завершить платёж
- `refund()` — возврат платежа
- `format_card()` — карточка платежа

**Свойства:**
- `amount_display` — "150.00 UAH"
- `status_display` — "✅ Завершен"
- `service_display` — "🚗 Парковка"
- `is_completed` — завершён ли
- `is_pending` — ожидает ли

## Альтернативы

| Альтернатива | Плюсы | Минусы |
|--------------|-------|--------|
| **Оставить как есть** | Нет работы | Нет развития, код не тестируемый |
| **Одна модель для всех** | Просто | Нарушает принцип единственной ответственности |
| **Микросервисы** | Масштабируемость | Избыточно для текущих задач |
| **Выбранное решение** | Чёткое разделение, тестируемость | Требует времени на реализацию |

---

## Последствия

### Положительные

1. **Тестируемость** — каждая модель может быть протестирована изолированно
2. **Понятность** — код отражает предметную область
3. **Развитие** — новые функции добавляются в соответствующие модели
4. **Безопасность** — старый код не трогается

### Отрицательные

1. **Дублирование** — пока идёт переход, часть логики остаётся в старом коде
2. **Время** — требуется постепенная миграция
