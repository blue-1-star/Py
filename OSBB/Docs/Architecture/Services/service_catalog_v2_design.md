# service_catalog v2 design

Дата: 2026-06-28  
Статус: проект архитектурного решения  
Контекст: OSBB / Live Services Sandbox / система управления задолженностью

---

## 1. Назначение документа

Этот документ фиксирует проект развития таблицы `service_catalog` до версии v2.

Цель — не создавать отдельный параллельный справочник услуг и не плодить отдельную таблицу правил задолженности без необходимости.  
Политика доступа к сервисам должна развиваться поверх уже существующих `service_code`.

---

## 2. Текущее состояние

Таблица `service_catalog` уже является центральным справочником услуг OSBB.

В текущей live-services sandbox базе она содержит:

- 16 строк;
- 16 колонок;
- 5 индексов;
- 0 foreign keys;
- 1900 найденных ссылок в коде;
- 16 service_code.

Существующие признаки уже позволяют классифицировать услуги:

- `service_code`
- `service_group`
- `service_name`
- `service_type`
- `category`
- `is_monthly`
- `is_fundraising`
- `is_commercial`
- `is_access_control`
- `is_cash_collectable`
- `is_active`

Особенно важны уже существующие поля:

- `is_access_control` — услуга относится к контролю доступа;
- `is_cash_collectable` — услуга может приниматься через кассу;
- `is_monthly` — регулярная ежемесячная услуга;
- `is_fundraising` — целевой сбор;
- `is_commercial` — коммерческая услуга.

## 4. Архитектурное решение

### Решение 1

`service_catalog` остаётся главным справочником услуг.

Не создаём второй справочник:

- действий;
- сервисов;
- ограничений;
- заказов;
- доступов.

### Решение 2

Политика задолженности v1 добавляется как свойства услуги в `service_catalog`.

### Решение 3

Сложная таблица правил `debt_policy_rules` пока не внедряется.

Она может появиться позже, если станет нужна связь many-to-many:

```text
долг по одному service_code
→ блокирует несколько разных target service_code
```

Но для текущего этапа это преждевременно.

---

## 5. Что добавляем в service_catalog v2

Минимальный набор полей v2:

| Поле | Тип | Назначение |
|---|---:|---|
| `requires_no_debt` | INTEGER DEFAULT 0 | Услуга требует отсутствия блокирующей задолженности |
| `debt_check_scope` | TEXT DEFAULT 'PARKING' | Какие долги проверять |
| `debt_block_mode` | TEXT DEFAULT 'NONE' | Режим ограничения: NONE / SOFT / HARD |
| `debt_block_message` | TEXT | Сообщение пользователю / оператору |
| `manual_review_required` | INTEGER DEFAULT 0 | Требуется ручное решение оператора |
| `policy_updated_at` | TEXT | Когда политика услуги была изменена |
| `policy_updated_by` | TEXT | Кто изменил политику |

---

## 6. Значения debt_block_mode

### NONE

Услуга не блокируется задолженностью.

Пример:

- `IMPROVEMENT`
- `HISTORICAL_UNCLASSIFIED`

### SOFT

Есть предупреждение о задолженности, но оператор может продолжить.

Пример:

- перепрошивка собственного пульта;
- изменение телефонного номера;
- спорные случаи.

### HARD

Услуга недоступна при наличии блокирующей задолженности.

Пример:

- заказ нового пульта;
- выдача нового пульта;
- подключение телефонного доступа.

---

## 7. Значения debt_check_scope

Для v1 используем простую строку-код:

| Значение | Что проверять |
|---|---|
| `NONE` | ничего не проверять |
| `PARKING` | долги по `PARKING_DAY` и `PARKING_NIGHT` |
| `ACCESS` | долги по услугам доступа |
| `ALL_CASH_COLLECTABLE` | все кассовые начисления |
| `MANUAL` | только ручной контроль |

Для текущего запуска достаточно `PARKING`.

## 9. Что сознательно не добавляем в v2

Не добавляем:

- `block_order`
- `block_issue`
- `block_activation`

Причина: это уже не свойство услуги, а свойство конкретного бизнес-действия.

Одна и та же услуга может иметь разные операции:

- заказ;
- выдача;
- активация;
- перепрошивка;
- повторное подключение;
- изменение данных.

Если сейчас добавить все такие флаги в `service_catalog`, таблица начнёт разрастаться. Для v1 достаточно:

```text
requires_no_debt + debt_block_mode + manual_review_required
```

А конкретный обработчик уже интерпретирует это в своём контексте.

## 10. Будущая функция проверки

Будущий единый интерфейс:

```python
result = check_service_allowed(
    apartment_number="89",
    service_code="TEST_REMOTE_NEW",
    action="order",
)
```

Пример результата:

```python
{
    "allowed": False,
    "mode": "HARD",
    "debt_total": 3000.0,
    "debt_services": ["PARKING_NIGHT"],
    "message": "По квартире есть задолженность за парковку. Заказ нового пульта временно недоступен."
}
```

## 11. Как это связано с кассой

Эта система не заменяет кассовый журнал.

Она создаёт правило:

```text
есть задолженность
→ услуга ограничена
```

Кассовый журнал должен обеспечить обратный процесс:

```text
оплата зарегистрирована
→ задолженность исчезла
→ ограничение снято автоматически
```

Поэтому после v2 `service_catalog` следующим производственным приоритетом остаётся рабочее место кассира.

## 14. Архитектурное заключение

`service_catalog` должен стать не только справочником начисляемых услуг, но и центральной точкой описания сервисной политики ОСББ.

Для версии v2 не создаём отдельную таблицу правил задолженности.  
Расширяем существующий каталог минимальными полями политики.

Критерий готовности v2:

```text
оператор может изменить политику услуги без изменения кода,
а бот использует эту политику при заказе/выдаче сервисов.
```

## 15. ADR-кандидат

ADR-0001:

```text
Решение:
Политика задолженности v1 реализуется через расширение service_catalog,
а не через отдельную таблицу debt_policy_rules.

Причина:
service_catalog уже является центральным справочником услуг,
активно используется в коде и содержит признаки классификации услуг.

Последствие:
новые сервисы получают политику через service_code.
Если позже понадобится many-to-many логика, будет добавлена отдельная таблица правил.
```
