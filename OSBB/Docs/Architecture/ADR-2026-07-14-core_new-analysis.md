## Контекст

Проект OSBB развивался как Telegram-бот. Бизнес-логика (работа с жителями, автомобилями, платежами) оказалась жёстко вплетена в код интерфейса.

### Проблема до внедрения

1. **Смешение ответственности:**
   - `Bots/parking_bot.py` содержал прямые вызовы к БД (`get_resident_account()`, `link_resident_to_apartment()`) и бизнес-проверки (`is_admin_user()`, `has_guard_workspace_access()`).
   - `Bots/handlers/client_portal.py` содержал бизнес-логику формирования счетов (`_billing_data()`), создания запросов на привязку квартир (`_create_apartment_link_request()`), что нарушало принцип единственной ответственности.

2. **Сложность тестирования:**
   - Бизнес-правила можно было проверить только через запуск бота и эмуляцию Telegram-сообщений.
   - Отсутствовала возможность изолированного тестирования логики.

3. **Жёсткая связанность:**
   - Любое изменение в бизнес-логике требовало правки интерфейсного кода.
   - Добавление нового интерфейса (например, Web-админки) требовало копирования логики из бота.

4. **Разрастание `db_access.py`:**
   - Файл вырос до 157 KB и содержал не только CRUD-операции, но и бизнес-логику (например, `get_next_vehicle_for_review()` содержало алгоритм выбора следующего автомобиля для проверки).

---

## Альтернативы, которые не подошли

| Альтернатива | Причина отклонения |
|--------------|---------------------|
| Полный рефакторинг | Слишком рискованно для проекта в стадии опытной эксплуатации. |
| Оставить как есть | Дальнейшее развитие невозможно, новый функционал (касса, коммерция) требовал бы дублирования кода. |
| Микросервисы | Избыточно для текущих задач, увеличивает сложность инфраструктуры. |

---

## Решение: создание слоя `core_new`

Создан новый слой, который:

1. **Не трогает старый код** — он продолжает работать.
2. **Содержит чистую бизнес-логику** в `domain/`.
3. **Общается со старым кодом через адаптеры** (`adapters/`).
