# OSBB — Единая информационная система управления ОСББ

## 1. Введение: Зачем всё это?

Проект OSBB родился из реальной жизненной проблемы:  
**хаос на парковке и в финансах**.

Жильцы не могли припарковаться, кассиры теряли бумажки, а инициативная группа увязала в учёте. Вместо того чтобы плодить Google-таблицы, было принято решение создать **единую цифровую среду**, которая:

- Решает финансовый хаос.
- Автоматизирует работу охраны и кассы.
- Обеспечивает прозрачность для жителей.
- Даёт инструменты для управления домом без программистов.

---

## 2. Архитектурная философия

Проект строится вокруг **трёх столпов**:

1. **Единое ядро данных** — все подсистемы работают с одной БД.
2. **Разделение интерфейса и логики** — Telegram — это лишь окно. Бизнес-логика (касса, автомобили, жители) живёт отдельно.
3. **Постепенная эволюция** — старый код не ломается, новый строится рядом (`core_new`).

---

## 3. Основные подсистемы (что решают)

| Подсистема | Решаемая задача |
|------------|-----------------|
| **Реестр жителей** | Кто живёт в доме, статус верификации, привязка к Telegram. |
| **Автомобили** | Учёт авто, тарифы (Day/Night), жизненный цикл (кандидат → активен → архив). |
| **Касса (Cashbox)** | Приём платежей наличными/банком, печать чеков, работа с «недоавтомобилями» (без квартиры). |
| **Каталог услуг** | Единый справочник: парковка, пульты, благоустройство, телефонный доступ. |
| **Политика доступа** | Блокировка услуг при долгах (например, отключение телефона при задолженности за парковку). |
| **Коммерческие договоры** | Отдельная подсистема для арендаторов и юрлиц. |
| **Аналитика и отчёты** | Срезы по БД: должники, нестандартные номера, автомобили без тарифа, последние платежи. |

---

## 4. База данных (ядро системы)

- **Движок:** SQLite (до 100+ таблиц).
- **Принцип:** единая БД для всех подсистем.
- **Ключевые сущности:**

| Таблица | Назначение |
|---------|------------|
| `resident_accounts` | Жители (Telegram-аккаунты). |
| `apartments` | Квартиры и помещения. |
| `vehicles` | Автомобили с тарифами (Day/Night). |
| `vehicle_candidates` | «Недоавтомобили» (без квартиры). |
| `payments` | Платежи. |
| `charges` | Начисления. |
| `cashbox_operations` | Кассовые операции (приход/расход). |
| `service_catalog` | Справочник услуг. |
| `service_items` | Тарифы и цены на услуги. |
| `payment_allocations` | Разнесение платежей по начислениям. |

## 5. Интерфейс: Telegram-бот

**Бот** — это не просто «меню с кнопками». Это **рабочее место** для разных ролей:

| Роль | Возможности |
|------|-------------|
| **Житель** | Привязка квартиры, просмотр авто, оплата, заказ пультов. |
| **Охрана (касса)** | Приём платежей, выдача чеков, работа с «недоавто». |
| **Оператор** | Верификация жителей, управление заявками на пульты. |
| **Администратор** | Управление услугами, тарифами, правами доступа. |

## 6. Дорожная карта (ROADMAP)

### ✅ Уже работает

- Реестр жителей и квартир.
- Учёт автомобилей с тарифами.
- Касса с поддержкой «недоавтомобилей».
- Единый каталог услуг.
- Система ролей и прав доступа.

### 🔄 В разработке (активно)

- **Интерфейс для подтверждения кандидатов** (превращение `vehicle_candidates` в автомобиль).
- **Умная касса** — подстановка суммы из `service_items` (вместо зашитых 500/300).
- **Политика доступа** — автоматическая блокировка услуг при долгах.

### 🚀 Планы (P1–P4)

| Приоритет | Задача |
|-----------|--------|
| **P1** | Рефакторинг идентичности жителя (разделение ролей). |
| **P2** | Переход на новую архитектуру `core_new` (выполнен). |
| **P3** | Умная касса с динамическими тарифами и расчётом периода. |
| **P4** | Полная поддержка коммерческих договоров и аренды. |

## 7. Ключевые архитектурные решения

| Решение | Зачем |
|---------|-------|
| **Слой `core_new`** | Отделить бизнес-логику от интерфейса (Telegram). |
| **`vehicle_candidates`** | Принимать платежи без квартиры. |
| **`service_catalog` + `service_items`** | Гибкое управление тарифами без пересборки кода. |
| **Адаптеры (`db_adapter`)** | Мост между старым кодом и новой архитектурой. |
| **Миграции (`migrations/`)** | Безопасные изменения схемы БД с возможностью отката. |
